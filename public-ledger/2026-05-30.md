## treestock: fixed "Sort: Relevance" being identical to "Sort: Name"

Benedict noticed that on treestock.com.au the "Relevance" sort produced the
same order as "Name: A-Z", especially with no search query and no filters.

**Root cause** (`tools/scrapers/build-dashboard.py`, client-side sort): the
relevance score is only computed when there is a search query (it ranks by how
early the first search term appears in the product title). With no query there
was nothing to score against, so the code fell through to a plain
alphabetical-by-title sort, which is exactly what "Name: A-Z" does. The two
options were functionally identical on the default view.

**Fix shipped:** Relevance now surfaces in-stock items first, then A-Z within
each group. When there is a search query, the title-match score is the primary
key and in-stock / A-Z are tiebreaks, so an in-stock match never sits below an
out-of-stock one at the same score. Committed (1f9b9a8), pushed, deployed, and
verified live.

**Follow-up:** Benedict's preferred default is "newest in stock first". That
needs a recency signal per product (when it first appeared / came back in
stock), which the dashboard data does not currently carry. It can be computed
at build time from the dated snapshot history (the same source
`build_recent_highlights` already scans). Tracked as the next iteration.

Note: tool-output rendering in this working session was intermittently
collapsing multi-line output, so the change was made and verified through
compact single-line checks (parse OK, brace balance OK, 41 unit tests pass,
live marker present).

Benedict asked how to convert more search visitors into back-in-stock
watchers. Investigation (Plausible + the watch DB) showed recent signups are
organic-search driven, landing on species and variety pages, but the watch
affordance lived almost entirely on individual /variety/ pages, and the
default "in stock only" filter on the homepage search hid the out-of-stock
items that are the prime watch targets.

**Shipped (Phase 1, search results)** in `tools/scrapers/build-dashboard.py`:

- The "In stock only" filter now defaults OFF, so out-of-stock items show up

- Each out-of-stock result row now shows a "Notify me when it's back in stock"

  the watch form). In-stock rows stay clean.
- Each product now carries its variety slug, derived from the existing,
  already-tested `product_variety_slug()` helper in `cultivar_parsing.py`, so
  the link always matches the built variety page (no slug drift, no 404s).

JS passes node --check, homepage returns 200, a sampled variety page returns
200, and 2,129 out-of-stock products now expose a notify link where they were
previously hidden. Cross-checked that every variety slug embedded in search has
a matching built /variety/ page (0 would-be 404s). 70 unit tests pass.
Committed and merged to main (867d31e), pushed, deployed.

## treestock: variety-aware notify links (per-nursery vs per-variety stock)

Benedict spotted the conceptual flaw: the back-in-stock watch is per-variety
across all nurseries (send_variety_alerts only fires when a variety goes from
zero-available-everywhere to available), but the per-row notify link I first
shipped keyed off the row's own per-nursery stock. A row out of stock at one
nursery but in stock at another offered a watch that would never usefully fire.

**Fix shipped and deployed:** compute variety-level availability once across all
nurseries (VARIETY_IN_STOCK), then branch each out-of-stock row:
- variety out of stock everywhere (1653 live rows) -> "Notify me when it's back
  in stock"
- variety in stock at another nursery (476 live rows) -> "In stock elsewhere ->"
Both link to the variety page (lists in-stock nurseries + watch form), so the
in-stock-elsewhere rows lead to a buy instead of a dead-end OOS nursery page.

**Process note (mistake + recovery):** the edit adding the VARIETY_IN_STOCK
declaration silently failed while the edit USING it succeeded, and the broken
commit was pushed to main. node --check passes undefined references (syntax
only) so it slipped through. Caught before any deploy reached production (the
server never pulled it). Fixed in d6223d6 and re-verified by EXECUTING the
render loop against real product data in Node (zero throws across all rows),
not just syntax-checking. Lesson: for the embedded dashboard JS, run the actual
render path, not node --check alone.

**Known gap (now closed, see below):** 1335 out-of-stock rows whose titles did
not parse to a variety slug showed no link (leading-quote titles like
'"Alstonville" Finger Lime', multigrafts, and noisy titles).

## treestock: relaxed cultivar parser to widen variety-page coverage

Benedict, testing with Sort: Name A-Z, saw the top out-of-stock rows (the
'"Alstonville" / "Maleny Red" Finger Lime' products) had no notify link.
Curly-quote titles sort to the top of A-Z, and those were exactly the rows
parse_cultivar could not handle, so the whole visible top of the list was
no-link rows. Investigation showed ~2,600 titles unparsed, bucketed by reason
(leading-quote, multigraft, parenthetical/size noise, category prefixes).

Built an additive "relaxed" pass in the shared cultivar_parsing.py that runs
ONLY when the strict parser returns None (strict behaviour and the 70 pinned
tests unchanged). It strips listing noise then re-finds a CANONICAL species
(not cultivar synonyms, so "Bacon Avocado" -> Avocado/Bacon), parses multigraft
trees as one sorted combo variety, and handles leading-quote titles, with a
high-precision denylist for aromatic/lookalike confusables (Lemon Myrtle, Bay,
Native Mulberry, Fiddle Leaf Fig, Brazilian/Cedar Cherry).

Decisions made with Benedict along the way: multigraft = one combo variety
(not split per cultivar); precision over recall (deny aromatics rather than
mis-label them); in-stock-elsewhere rows link to the variety page to buy.

Process: wrote tests first, then audited against the live product set BEFORE
deploying. +1132 titles now parse (3634 -> 4766), ZERO strict regressions, and
the only slug merges were the same cultivar via different title formats
correctly collapsing (cross-nursery dedup). 74 tests pass.

Also fixed a latent broken link: the dashboard emitted vs=apple-yates for
"Yates Apple (dwarf)" but build_variety_pages filters "yates" (the chemical
brand) out of NON_PLANT_KEYWORDS and never built the page. Imported that
denylist into the dashboard so the two can't drift (single source of truth).

Deployed: variety pages 3033 -> 4592 built, dashboard rebuilt, JS verified,
0 broken variety links live. OOS rows with no notify link dropped from ~1335
to 787 (the remainder are bare species names and genuinely non-cultivar items).
Confirmed in Chrome under Sort: Name A-Z that the finger-lime rows now show the
notify link. Three commits, merged to main, deployed.
